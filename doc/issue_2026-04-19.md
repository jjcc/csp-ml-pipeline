# Backfill Pipeline Issues

Investigation of `scripts/backfill_rolling_windows.py` and the data preparation pipeline
after running the 2025-04-28 → 2025-08-04 backfill batch.

---

## Bug 1 (Critical): `score_window_trade_days` does not match actual data density

**File**: `scripts/backfill_rolling_windows.py` — `infer_score_window_trade_days()`

**What it does**: Infers the score window size by counting calendar business days between
`dataset.events_start_date` and `dataset.events_end_date` in the base config:

```python
trade_days = len(pd.bdate_range(start, end))  # Oct 27 → Nov 7 = 10
```

**The problem**: `filter_by_dte` (called from `a01_build_features.py`) previously kept only
DTE=4 trades captured on Monday and DTE=3 trades captured on Tuesday. This reduces the
observed data to **2 days per calendar week**, not 5. Requesting a chunk of 10 "observed
trade days" therefore requires 5 calendar weeks of data, not 2 weeks as the spec intends.

**Observed effect**: After the 14-week warmup ends on 2025-08-04, only 4 Mon/Tue dates
exist in the labeled CSV (Aug 4, 5, 11, 12). Since 4 < 10, only **one partial window**
was generated instead of multiple complete 2-week windows. The entire backtest rests on
a single 4-day window.

**Related fix**: `filter_by_dte` has been changed to `DTE <= 8` (all weekdays), which
brings observed density back to ~5 days/week and makes `score_window_trade_days=10` align
with a 2-calendar-week window. Re-running the full prep pass is required.

## Bug 2 (Minor): `build_prep_config` receives wrong `overall_end` when `end_date` is None

**File**: `scripts/backfill_rolling_windows.py` — `main()`

**Code**:
```python
prep_cfg = build_prep_config(
    base_cfg=base_cfg,
    overall_start=history_start_date,
    overall_end=end_date if end_date is not None else history_start_date,  # ← wrong
)
```

When `end_date` is not provided, `overall_end` is set to `history_start_date` (2025-04-28).
`build_prep_config` sets `cutoff_date = overall_end + 21 days = 2025-05-19`, which would
prevent labeling of almost all trades. This is immediately overridden by the option-filename
inference block below, so the run does not fail today. However, if that override block is
ever removed or reorganised the prep pass will silently build a near-empty labeled CSV.

**Fix**: Pass the inferred end date explicitly before calling `build_prep_config`, or use
a sentinel value that makes the incorrect fallback obvious.

## Issue 3: Insufficient scoring data after the 14-week warmup

Not a code bug, but explains why only one window was produced.

The labeled CSV spans 2025-04-28 → 2025-08-12 (30 Mon/Tue trade dates with the old
DTE filter). The 14-week warmup consumes 26 of them (up to 2025-07-29). Only 4 dates
remain for scoring (Aug 4, 5, 11, 12), which is less than the configured
`score_window_trade_days=10`.

After switching to `DTE <= 8`, the labeled CSV will be substantially denser and a
longer end date should be used so that multiple complete score windows exist after
the warmup.

## Issue 4: Training OOF ranking is inverted — top-10% by `p_bin3` has **negative** mean return

**Observed in**: `output/backfill/rolling_window_summary.csv`

```
winner_train_top10pct_mean_target   = -2.998
winner_train_bottom10pct_mean_target = -4.597
winner_train_top_minus_bottom_spread =  1.599
```

The spread is positive (model does rank trades in some order), but both the top and
bottom deciles show negative mean `return_mon`. The model trained on the April–July 2025
window (Liberation Day tariff-crash + recovery) learned patterns that are not predictive
of absolute return in either direction. In OOF evaluation the highest-confidence "bin 3"
trades underperformed the training mean.

This is not a code defect but a data-quality / regime problem:
- The training period is dominated by an unusual crash-recovery dynamic.
- With only ~26 unique Mon/Tue trade dates (≈ 8 000 rows) across 14 weeks, the
  training set is small for a 4-class multiclass model with 5-fold TimeSeriesSplit.
- Expanding to `DTE <= 8` will increase training volume by roughly 5×, which should
  stabilise the OOF estimates.

## Issue 5: Auto-calibration collapses to threshold = 0.9996, selecting 1 trade out of 1 499

**Observed in**: `output/backfill/rolling_window_summary.csv`

```
winner_score_threshold          = 0.9996
winner_score_predicted_winners  = 1
winner_score_coverage           = 0.00067
tail_score_tails_flagged        = 1499   (all trades)
tail_score_safe_fraction        = 0.0
```

The score window (Aug 4–12, 2025) had an 87.5% binary win rate (`return_mon > 0`).
The configured `target_precision = 0.90` is barely above that baseline. Because the
model's `p_bin3` score provides no lift over 87.5% in this regime, `pick_threshold_auto`
scans upward until it reaches the maximum threshold — selecting just 1 trade.

The Aug 4–12 window also coincides with the Nikkei −12% crash on Aug 5. A model trained
on the preceding recovery period has no signal to distinguish the relative 12.5% of losers
from winners in such extreme conditions.

Contributing code factor: `pick_threshold_auto` only checks every 5th candidate threshold
(`if idx % 5 != 0: continue`), so it can jump past the first valid threshold and land on
a much stricter one. For a dense probability distribution this coarse sampling can
exaggerate the threshold.

## Recommended Next Steps

1. ✅ **Re-run the full prep pass** (`a01 → a04`) with the updated `DTE <= 8` filter
   (`service/preprocess.py`) to build a denser labeled CSV.
2. ✅ **Use a longer backfill end date** — fixed as a side-effect of step 3: end date is
   now inferred from the latest option snapshot file before the prep pass, so the prep
   automatically covers all available data.
3. ✅ **Fix Bug 2** (`scripts/backfill_rolling_windows.py`): end date is now resolved
   from option files **before** `build_prep_config` is called, eliminating the incorrect
   `history_start_date` fallback and the fragile post-call override.
4. ✅ **Fix `pick_threshold_auto` sampling** (`service/utils.py`): removed the
   `if idx % 5 != 0: continue` stride so every candidate threshold is evaluated.
   Previously the coarse scan could skip the first valid threshold and land on a
   far stricter one.
5. ✅ **Flag anomalous score windows** (`scripts/backfill_rolling_windows.py`): the
   rolling summary CSV now includes six extra columns per window:
   - `train_win_rate` / `score_win_rate` — fraction of trades with `return_mon > 0`
   - `train_mean_return_mon` / `score_mean_return_mon` — mean monthly return
   - `win_rate_gap` — absolute difference between score and train win rates
   - `regime_flag` — `True` when `win_rate_gap > 0.10` (score period diverged from
     training by more than 10 pp), indicating the window should be interpreted
     cautiously.
